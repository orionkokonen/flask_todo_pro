0. 全体マップ（最初に頭に入れる地図）
このアプリでパスワードに関係するファイルは3つだけです。

ファイル	役割
app/models.py:133-146	set_password / check_password（ハッシュ化と照合の本体）
app/auth/routes.py:29	タイミング攻撃対策のダミーハッシュ
app/forms.py:78-119	パスワード強度チェック（最低文字数・文字種）
ここを押さえれば「パスワードはどう守られているか？」に答えられます。

1. 大前提：なぜ平文（生のパスワード）で保存してはいけないのか
面接Q：「パスワードはDBにそのまま保存していますか？」
答え方：

平文では保存していません。scrypt という方式でハッシュ化した値だけを password_hash カラムに保存しています。

なぜ？（初心者向けの理由）
DBが万が一漏れたとき（不正アクセス・内部関係者の流出など）、平文だと全ユーザーのパスワードがそのまま盗まれるからです。

しかも怖いのは、人間は 同じパスワードを他のサイトでも使い回す こと。1つのアプリから漏れると、銀行・SNS・メール…全部ドミノ倒しになる。これを防ぐのが第一目的です。

2. ハッシュ化とは何か（暗号化との違い）
専門用語：「ハッシュ化」
入力（パスワード）を、元に戻せない決まった長さの文字列に変換する処理。

暗号化	ハッシュ化
元に戻せる？	✅ 鍵があれば戻せる	❌ 一方通行（戻せない）
用途	通信の保護など	パスワード保管など
ポイント：戻せない＝盗まれても平文を復元できない。

確認はどうやるの？
戻せないのに、ログイン時にどう一致を確認するのか？

→ 「入力されたパスワードを、同じ方法でもう一度ハッシュ化して、保存済みハッシュと比べる」 だけ。
ハッシュ関数は「同じ入力からは必ず同じ出力」になる性質があるので、これでOK。

app/models.py:141-146


def check_password(self, password: str) -> bool:
    return check_password_hash(self.password_hash, password)
3. なぜ「scrypt」なのか（MD5やSHA-256じゃダメな理由）
ここが一番面接で聞かれやすい部分です。

面接Q：「ハッシュ方式に scrypt を選んだ理由は？」
答え方（テンプレ）：

MD5 や SHA-256 は 高速すぎて、攻撃者がGPUで1秒間に何十億回も総当たりできてしまいます。scrypt はわざと 計算コストが重く、メモリも大量に使う ように設計されているため、総当たり攻撃を実用的でない速度まで遅くできます。

噛み砕いた説明
MD5 / SHA-256：もともと「ファイルが改ざんされてないか」を高速にチェックするためのもの。速いことが目的。 → パスワード保管に使うと、攻撃者にも有利。
bcrypt / scrypt / Argon2：パスワード保管専用に作られた、わざと遅いハッシュ。 → 1回の計算が0.1秒くらいかかる。攻撃者が10億回試そうとすると現実的な時間で終わらない。
scryptはさらに メモリも大量に消費 するので、GPUによる並列攻撃にも強い。
コードで該当箇所
app/models.py:133-139


def set_password(self, password: str) -> None:
    self.password_hash = generate_password_hash(password, method="scrypt")
method="scrypt" を明示しているのが大事。コメントにも書いてある通り、ライブラリ更新時に勝手に方式が変わるのを防げる。

4. ソルト（salt）— 同じパスワードでも違うハッシュにする仕組み
werkzeug の generate_password_hash は、内部で自動的に ソルト を付けてくれます。これも面接で聞かれやすい。

専門用語：「ソルト（salt）」
ハッシュ化するときに、パスワードに混ぜるランダムな文字列。

なぜ必要？
ソルトがないと：

ユーザーAとユーザーBが偶然同じ「password123」を使うと、ハッシュも完全に同じ。
攻撃者は「よくあるパスワード一覧をハッシュ化した辞書（レインボーテーブル）」を事前に作っておき、DBが漏れたら一瞬で照合できる。
ソルトがあると：

各ユーザーに 別々のランダム文字列 を混ぜてからハッシュ化する。
同じ「password123」でも、ユーザーごとに全く違うハッシュになる。
レインボーテーブル攻撃が事実上効かなくなる。
コードでは？
generate_password_hash が 自動でソルトを生成して、ハッシュ値の中に一緒に埋め込んで くれます。なので保存するのは password_hash の1カラムだけでOK。
保存される文字列はだいたいこんな形：


scrypt:32768:8:1$<ランダムなsalt>$<ハッシュ本体>
照合時の check_password_hash も、この文字列からアルゴリズム名・ソルト・ハッシュ本体を読み取って、自動的に同じ手順で再計算してくれる。便利。

5. タイミング攻撃 — 地味だけど面接でウケる話
app/auth/routes.py:26-29 の謎の定数：


_TIMING_EQUALIZATION_HASH = generate_password_hash("dummy-timing-pad", method="scrypt")
これ、面接で聞かれたら得点源になります。

専門用語：「タイミング攻撃」
処理にかかった時間の差から、内部の状態を推測する攻撃。

このアプリで何が起きうるか
愚直に書くと：

ユーザーが存在する → ハッシュ照合する（重い計算、例：100ms かかる）
ユーザーが存在しない → 何もせず即「失敗」を返す（1ms で終わる）
→ 攻撃者は応答時間を測るだけで、「このユーザー名は登録されている／されていない」を見分けられる。

これを使うとユーザー名一覧を作られてしまう（後の総当たりやスパムの下準備になる）。

このコードの対策
app/auth/routes.py:163-170


if user is None:
    # 結果は使わず、計算時間だけそろえる
    check_password_hash(_TIMING_EQUALIZATION_HASH, form.password.data)
else:
    password_matches = user.check_password(form.password.data)
ユーザーが見つからなかった場合でも、ダミーのハッシュに対して同じ重さの計算を1回回す。
こうすると応答時間が「あった場合」と「なかった場合」でほぼ揃い、ユーザー名の存在を時間差から見破れなくなる。

check_password_hash 自身もタイミング攻撃対策済み
app/models.py:141-146 のコメント：

比較にかかる時間を一定にする処理（タイミング攻撃対策）は werkzeug 内部で実施される。

文字列比較も == で1文字ずつ早期リターンしてしまうと「先頭何文字が当たっているか」を時間差で見破れる。werkzeug は内部で 定数時間比較（どの位置で違っても同じ時間かける比較）を使っている。

6. パスワード強度チェック（入る前の門番）
ハッシュ化が「もしDBが漏れても守る」防御なら、強度ポリシーは「そもそも弱いパスワードを登録させない」防御。両方あって初めて強い。

app/forms.py:78-119

設定で切り替えられるようになっている：

PASSWORD_MIN_LENGTH：最低文字数（デフォルト12）
PASSWORD_REQUIRE_UPPER / LOWER / DIGIT / SYMBOL：文字種要件
面接で言えると強い一言
パスワードポリシーをコード直書きではなく config で外出し しているので、運用中にポリシーを強くしたいときも、コード変更なしで切り替えられる設計にしました。

7. 面接想定Q&A まとめ（暗記用）
Q1. パスワードはどう保存していますか？
平文では保存していません。werkzeug.security.generate_password_hash で scrypt によりハッシュ化し、User.password_hash カラムに保存しています。ソルトはライブラリが自動生成し、ハッシュ文字列に埋め込まれます。

Q2. なぜ scrypt を選んだのですか？
MD5 や SHA-256 は計算が速すぎて、漏洩時に総当たりされやすい。scrypt はわざと計算コストとメモリ使用量を高くしているため、ブルートフォースを実用的に困難にできるからです。

Q3. ソルトは何のためにありますか？
同じパスワードでもユーザーごとに違うハッシュ値にし、レインボーテーブル攻撃（事前計算した辞書による攻撃）を無効化するためです。

Q4. ログイン処理で気をつけている点は？（Q2と被らない範囲で）
タイミング攻撃対策として、ユーザー名が存在しない場合でもダミーのハッシュ照合を1回走らせ、応答時間からユーザーの有無が漏れないようにしています。

Q5. 平文パスワードはどこかでログに残っていませんか？
残していません。フォームから受け取った form.password.data は set_password か check_password にしか渡しておらず、ログ出力には username と ip だけを残しています（app/auth/routes.py:179-202）。

8. 一言サマリ（30秒で言うなら）
パスワードは平文ではなく scrypt によるハッシュ化＋自動ソルト で保存しています。scrypt は意図的に計算が重いハッシュなので、DBが漏れても総当たりが現実的に困難です。さらに、ログイン時にはユーザーが存在しなくてもダミーのハッシュ照合を走らせる タイミング攻撃対策 を入れ、werkzeug 側の定数時間比較と合わせて、応答時間からの情報漏えいも防いでいます。

これを言えれば面接ではお釣りが来ます。
